# 超值限时组合包购买规划规则

本文档记录当前规划器采用的规则假设，适用于 `PackCalculatorView.vue` 中的购买方案规划功能。

路径计算算法见 `path-planning-algorithm.md`。

## 档位

超值限时组合包的档位由付费钻数量决定。项目数据用日元价格表示，同一档位的对应关系如下：

| 价格 | 付费钻 |
| ---: | ---: |
| 160 | 80 |
| 650 | 325 |
| 1000 | 500 |
| 1500 | 750 |
| 3000 | 1500 |
| 6000 | 3000 |
| 11800 | 5900 |

初始触发的限时包通常是 80 付费钻档，也就是价格 160 档。玩家当前已经处在更高档时，可以在规划器中把“当前档位”改成对应价格。

## 升降档

- 触发并购买至少 1 个限时包后，下个结算批次升 1 档。
- 触发但没有购买，2 小时后超时，下个结算批次降 1 档。
- 档位不会低于最低档，也不会高于最高档。

## 卡包批次

当 2 小时内连续触发多个限时包时，这些礼包视作同一批：

- 在最后一个礼包消失前，批内至少购买 1 个，则下一批升 1 档。
- 批内其他礼包不买，不会额外导致降档。
- 批内全部不买，则下一批降 1 档。
- 规划器用“每批最多触发数”模拟 2 小时内可连续触发多少个礼包。

## 触发节奏

- 主线和无穷塔可以随推关连续触发，适合把“每批最多触发数”设为大于 1。
- 单个属性塔每天只能爬 10 层，不能像无穷塔一样持续触发，适合设为 1 或当天实际可触发数。
- 玩家等级提升较慢，通常适合设为 1。
- 不同类别的限时包可以同时触发；主线、玩家等级、无穷塔、各属性塔在规划器中视作独立触发源。
- 规划器会先按每个触发源独立生成批次，再把各触发源的第 N 批合并为同一个全局结算批次，统一应用升降档规则。

页面字段含义：

- 预算（日元）：本次可用于购买限时组合包的金额上限。
- 当前礼包档位：现在已经处于的限时组合包档位。
- 当前层/等级：该触发源目前已经达到的位置；规划器只会考虑大于该值的触发点。
- 计划推到：本次规划中该触发源最多推进到的位置。
- 2小时内最多触发：该触发源在一个卡包批次内最多能连续触发多少个限时包。

## 全属性塔抵达

“全属性塔抵达”不是独立填写的触发源，而是由蓝塔、红塔、翠塔、黄塔自动派生：

```
全属性塔进度 = min(蓝塔层数, 红塔层数, 翠塔层数, 黄塔层数)
```

只有四个属性塔都在规划器中启用，并且四塔共同跨过某个“全属性塔抵达”礼包层级时，该礼包才会进入对应批次。

派生批次按四塔中最慢达到该层的塔计算：

```
触发批次 = max(蓝塔达到该层的批次, 红塔达到该层的批次, 翠塔达到该层的批次, 黄塔达到该层的批次)
```

因此无需为“全属性塔抵达”单独填写起始层、目标层或每批触发数。

## 最优路径目标

规划器在预算内最大化购买礼包的总评分价值。礼包价值继续使用 `items-calc-formula.md` 中定义的 CE 评分逻辑；同一批同一档内，如果可买多个礼包，会优先选择价值更高的礼包。

若预算用不完，是因为继续购买会超出预算，或后续在当前档位没有可购买礼包。

规划器不会在输入变化时自动重算；需要用户点击“计算购买方案”。这样可以避免多触发源全部启用时，输入框每次变化都触发动态规划。

为了控制状态规模，动态规划每个批次后会按档位做状态裁剪：保留预算-价值维度上的 Pareto 状态，并限制每个档位最多保留一批高价值候选路径。

## 多方案展示

页面会同时生成多种方案，供玩家按偏好切换：

- 价值最优：用动态规划在预算内寻找总评分价值最高的路径。
- 次优方案：同一动态规划中保留候选路径，展示价值排名第二的路径。
- 只买小包：只在最低档购买，升档后等待超时掉回最低档。
- 冲最大包：先每批买 1 个用于升档，到最高档后尽量购买高档礼包。

这些方案共用同一套触发批次和升降档规则，只是购买动作选择策略不同。

## 属性塔原始开放规则

属性塔本身存在真实日期限制：

| 星期 | 开放属性塔 |
| --- | --- |
| 周一 | 蓝塔 |
| 周二 | 红塔 |
| 周三 | 翠塔 |
| 周四 | 黄塔 |
| 周五 | 蓝塔、红塔 |
| 周六 | 翠塔、黄塔 |
| 周日 | 蓝塔、红塔、翠塔、黄塔 |

每个单属性塔每天最多推进 10 层。若玩家在换日前后 2 小时内卡包，单个属性塔在同一个限时包批次内理论上最多可以跨两天推进：

```text
跨日同批最大推进 = 10 + 10 = 20 层
```

## 乐观卡包与属性塔拓扑

规划器默认采用“乐观卡包”假设：玩家会主动控制主线、无穷塔、属性塔等来源的触发时机，让可安排的礼包尽量落入同一个 2 小时卡包批次。

该假设不代表可以任意打乱属性塔事件顺序。属性塔仍按触发层级建立拓扑序列：

```text
全属性塔抵达 225
单属性塔 250（蓝/红/翠/黄可同批）
全属性塔抵达 275
单属性塔 300（蓝/红/翠/黄可同批）
...
```

全属性塔抵达由四塔最低层决定：

```text
全属性塔进度 = min(蓝塔层数, 红塔层数, 翠塔层数, 黄塔层数)
```

当该进度跨过全属性塔触发层时，才生成全属性塔抵达礼包。

属性塔相关礼包的相邻触发事件至少相隔 25 层：

- 全属性塔抵达与下一个单属性塔触发相隔 25 层。
- 单属性塔触发与下一个全属性塔抵达相隔 25 层。
- 同类事件相隔 50 层。

即使考虑换日前后 2 小时卡包，单塔最多也只能推进 20 层，仍无法跨过相邻属性塔礼包事件。因此当前规划器忽略跨日合批能力，并把蓝塔、红塔、翠塔、黄塔的“2小时内最多触发”固定为 1。

## 日期抽象原因

当前规划器将真实日期抽象掉，不要求用户填写开始日期或规划天数，原因如下：

1. 规划器采用乐观卡包假设，认为玩家会主动卡塔，把属性塔尽量停在触发点前，并在可同步窗口集中触发。
2. 当前属性塔相关礼包事件的相邻间隔至少是 25 层：
   - 全属性塔抵达 225 -> 单属性塔 250，相隔 25 层。
   - 单属性塔 250 -> 全属性塔抵达 275，相隔 25 层。
   - 同类事件之间相隔 50 层。
3. 跨日同批最多推进 20 层，小于相邻事件的 25 层间隔，因此跨日不会让两个相邻属性塔礼包事件合并进同一个批次。
4. 真实星期只影响玩家准备卡点的过程，不改变礼包事件的拓扑顺序。

因此算法只保留“属性塔事件拓扑序”，不模拟真实日历：

```text
全属性塔抵达 225
单属性塔 250（蓝/红/翠/黄可同批）
全属性塔抵达 275
单属性塔 300（蓝/红/翠/黄可同批）
...
```

该抽象依赖当前礼包触发数据。如果未来属性塔或全属性塔礼包触发间隔缩短到 20 层以内，需要重新评估是否必须引入真实日期/开放窗口模型。
