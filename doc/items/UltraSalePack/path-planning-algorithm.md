# 限时组合包路径计算算法

本文记录购买方案规划器的路径计算算法，对应实现：

- `src/engine/ultraSalePlanner.js`
- `src/views/PackCalculatorView.vue`
- `test/ultraSalePlanner.test.js`

规则假设见 `planning-rules.md`。

流程图和伪代码速览见 `path-planning-visual.md`。

## 目标

在用户给定的预算（日元）、当前礼包档位、各触发源当前进度与计划推进范围内，计算一条或多条购买路径。

核心目标是：

```text
在总花费 <= 预算的约束下，最大化购买礼包的总评分价值
```

评分价值来自礼包 CE 计算结果，继续复用 `items-calc-formula.md` 与 `packCalc.js` 的物品评分逻辑。

当两条路径总价值相同时，排序规则为：

1. 购买礼包数量更多者优先。
2. 若购买数量也相同，花费更少者优先。

## 数据建模

### 礼包档位

算法从礼包数据中提取可用价格档位并升序排列，例如：

```text
160, 650, 1000, 1500, 3000, 6000, 11800
```

状态中不直接保存价格，而是保存 `tierIndex`，即当前处于第几个档位。

购买动作会让 `tierIndex + 1`，不购买动作会让 `tierIndex - 1`。档位会被限制在最低档和最高档之间。

### 触发机会

每个触发点会被整理成一个 `opportunity`：

```text
触发源 + 触发进度 + 各价格档位可买的礼包
```

例如主线 `10-28`、无穷塔 `180`、蓝塔 `80` 都会成为独立触发机会。

同一个触发点可能有多个价格档位的礼包。规划到某一批时，只会根据当时的 `tierIndex` 取对应档位的礼包。

### 触发进度排序

不同来源的触发进度会转换成可比较的数字：

- 普通数字触发点直接使用数字。
- 主线 `章节-关卡` 由 `mainQuestProgress.js` 转成可比较的主线进度。

用户输入的主线当前进度和计划推到也使用同一套解析规则，因此可以填写 `13-28`，也可以直接填写游戏内连续关卡进度。其他来源通常填写数字层数或等级。

每个触发源只保留：

```text
当前进度 < 触发点 <= 计划推到
```

范围内的触发机会。

## 批次生成

每个触发源先独立生成批次。

`2小时内最多触发` 表示该来源在一个卡包批次内最多放入多少个触发机会。例如无穷塔从 100 推到 130，若每批为 3，则会按排序后的触发机会每 3 个切成一批。

不同触发源相互独立，但可以同时触发。因此全局批次按批次序号合并：

```text
全局第 1 批 = 主线第 1 批 + 等级第 1 批 + 无穷塔第 1 批 + 各属性塔第 1 批
全局第 2 批 = 主线第 2 批 + 等级第 2 批 + 无穷塔第 2 批 + 各属性塔第 2 批
...
```

每个全局批次统一应用一次升降档规则：

- 批内至少买 1 个礼包：下一批升 1 档。
- 批内全不买：下一批降 1 档。

## 全属性塔抵达

“全属性塔抵达”不是用户单独填写的触发源，而是从四个属性塔推导。

算法要求蓝塔、红塔、翠塔、黄塔四个来源都启用，才会尝试派生全属性塔礼包。

对于某个全属性塔触发层 `floor`：

```text
四塔共同进度 = min(蓝塔层数, 红塔层数, 翠塔层数, 黄塔层数)
```

只有四塔在规划范围内都能达到该层时，该触发机会才有效。

派生触发机会所属批次取四塔中最晚到达该层的批次：

```text
派生批次 = max(蓝塔到达批次, 红塔到达批次, 翠塔到达批次, 黄塔到达批次)
```

然后把该机会加入对应的全局批次。

## 动态规划

最优路径使用动态规划计算。

### 状态

每个状态表示处理完若干批次后的路径结果：

```js
{
  tierIndex,   // 下一批开始时的礼包档位
  spent,       // 已花费日元
  value,       // 已获得礼包总评分价值
  purchases,   // 已购买礼包数量
  steps        // 路径明细
}
```

状态表 key 为：

```text
tierIndex | spent
```

同一档位、同一花费下，可能保留少量不同路径，用于生成次优方案。

### 动作

对每个全局批次，根据当前档位找到批内可买礼包。

批内动作包括：

1. 全不买。
2. 买 1 个。
3. 买 2 个。
4. 以此类推，最多买完整个批次内当前档位存在的礼包。

批内可买礼包会先按评分价值从高到低排序。买多个时，算法只考虑“从高价值开始买前 N 个”的前缀动作。

这样可以避免枚举批内所有子集。因为在同一档位、同一价格下，预算成本随购买数量线性增加，买 N 个时选择价值最高的 N 个不会差于其他组合。

### 转移

对每个状态和每个动作，生成候选新状态：

```text
newSpent = spent + action.cost
```

若 `newSpent > budget`，丢弃该候选。

否则根据是否购买更新档位：

```text
action.bought ? tierIndex + 1 : tierIndex - 1
```

并累加价值、购买数量和路径明细。

### 剪枝

触发源很多时，状态数会快速膨胀。当前实现每处理完一个批次后执行剪枝。

剪枝分两步：

1. 按 `tierIndex` 分组。
2. 在每个档位内按花费从低到高扫描，只保留 Pareto 状态。

Pareto 状态含义：

```text
若某状态花费更高，价值却不高于已有状态，则它不可能成为更优路径，可以丢弃。
```

之后再按路径排序规则保留前 `maxStatesPerTier` 个状态。页面当前使用 `350`，引擎默认值也是 `350`，并设置最低有效值 `50`。

这是一种近似受控的动态规划：保留足够多的高价值候选，同时避免 7 个触发源全启用时浏览器卡死。

## 多方案生成

页面展示的方案分两类。

### 价值最优 / 次优方案

这两个方案来自同一次动态规划。

动态规划结束后，把所有保留状态按排序规则排列：

```text
总价值高 -> 购买数多 -> 花费少
```

取第 1 条作为“价值最优”，取第 2 条作为“次优方案”。

为了避免同一条路径重复出现，算法会用路径签名去重。签名由每批档位与购买触发点组成。

### 策略方案

策略方案不用动态规划搜索，而是按固定规则模拟同一批次列表。

“只买小包”：

- 只有当前档位是最低档时才买。
- 买入当前批内价值最高的 1 个最低档礼包。
- 升档后等待超时掉回最低档。

“冲最大包”：

- 未到最高档前，每批买 1 个价值最高礼包用于升档。
- 到最高档后，在预算内尽量购买当前批内的高档礼包。

策略方案与最优方案共享同一套触发批次、预算限制和升降档规则，只是动作选择方式不同。

## 路径展示压缩

计算结果中的每个 `step` 对应一个全局批次。

为了让页面更容易阅读，展示层会压缩连续不购买的批次：

```text
不买第 2 批 + 不买第 3 批 + 不买第 4 批
```

显示为：

```text
第 2-4 批，连续不买 ×3
```

购买批次不会合并，因为用户需要展开查看每个礼包的触发点和内容物。

压缩逻辑由 `compressUltraSalePlanSteps()` 处理，并有单元测试覆盖。

## 复杂度

设：

- `B` 为全局批次数。
- `S` 为剪枝后每批保留状态数。
- `A` 为单批可选动作数。
- `T` 为档位数。

动态规划主体近似为：

```text
O(B * S * A)
```

剪枝会按档位处理状态，实际状态上限受 `maxStatesPerTier * T` 控制。

单批动作数通常等于：

```text
1 + 当前批内当前档位可买礼包数
```

其中 `1` 是“全不买”动作。

当前主要性能瓶颈：

- 默认触发范围过大时，`B` 会直接变大；7 个来源全启用时，每一批都会推动一次 DP 状态转移。
- 单批合并多个来源时，`A` 会随当前档位可买礼包数上升。
- 开启常驻补累充后，每个“本批购买”候选还会额外分叉为“不补 / 补到下一累充档”，并需要计算常驻包组合。
- 结果展示需要保留完整路径；若每次去重都重新遍历完整 `steps`，长路径会放大开销。

已做的性能控制：

- 页面不会限制默认触发区间；批次数决定后续触发机会，缩小批次数会改变最优路径。
- 页面不会随输入自动重算，必须点击“计算购买方案”。
- 每批结束后按档位做 Pareto 剪枝，并限制 `maxStatesPerTier`。
- 补累充候选按累计花费缓存；同一累计花费下的下一累充档、剩余预算和常驻包组合相同，无需重复背包计算。
- 路径签名随状态增量保存，减少候选去重时反复扫描完整路径。

仍可继续优化的方向：

- 把动态规划移入 Web Worker，避免极端大范围计算阻塞主线程。
- 给页面增加“快速/精确”模式：快速模式可进一步降低 `maxStatesPerTier` 或只计算最优解，不生成次优方案。
- 在用户输入特别大的区间时显示批次数预估和耗时风险提示。
- 允许用户显式选择“仅用 80 钻石包补累充”作为快速近似模式；该模式会牺牲一部分预算最优性，因此不应替代默认精确补包。

## 每日累充与补包

礼包查询页仍复用 `packCalc.js` 的 `RechargeValue`，按 `PaidDiamonds * 1.2` 展示单个礼包的期望充值价值。

购买规划页使用实际每日累充规则。每日累充赠钻规则见 `../items-calc-formula.md`，在每日累计 12,000 付费钻之后，后续区间的边际赠钻倍率会下降。

因此购买路径中的充值补正是路径相关值：

```text
同一个礼包的实际充值补正
= 购买前后每日累计付费钻跨过的赠钻区间差值
```

规划器把礼包内容价值和充值补正拆开计算：

```text
actionValue = action.originalValue + actualRechargeValueDelta
```

其中 `actualRechargeValueDelta` 由 `src/engine/dailyRechargeBonus.js` 按当前路径已累计付费钻实时计算。

### 每日累充函数

`src/engine/dailyRechargeBonus.js` 提供：

```js
const DAILY_RECHARGE_BONUS_TIERS = [
  { paid: 100, bonus: 120 },
  { paid: 500, bonus: 480 },
  { paid: 1000, bonus: 600 },
  { paid: 2000, bonus: 1200 },
  { paid: 6000, bonus: 4800 },
  { paid: 12000, bonus: 7200 },
  { paid: 18000, bonus: 4200 },
  { paid: 24000, bonus: 5400 },
  { paid: 36000, bonus: 10800 }
]

cumulativeFreeDiamonds(paidDiamonds)
marginalFreeDiamonds(beforePaidDiamonds, addedPaidDiamonds)
unlockedRechargeTiers(beforePaidDiamonds, addedPaidDiamonds)
```

`cumulativeFreeDiamonds()` 返回同一天累计付费钻已获得的免费钻总数。`marginalFreeDiamonds()` 返回本次购买动作带来的边际赠钻：

```text
Bonus(after) - Bonus(before)
```

`unlockedRechargeTiers()` 用于页面展示本次购买或补包解锁了哪些累充档位。

### DP 状态

单日规划中，当前已花费日元可以换算为已累计付费钻：

```text
dailyPaidDiamonds = spent / 2
```

当前页面不要求填写具体日期，购买规划按“同一个每日累充周期内完成”计算。如果未来支持跨 0 点规划，则状态需要显式增加：

```js
{
  dayIndex,
  dailyPaidDiamonds
}
```

并在 0 点重置 `dailyPaidDiamonds`。

### 动作估值

批内动作预生成“买哪些包、花多少钱、基础内容价值、增加多少付费钻”，不在动作生成阶段写死最终价值。

```js
{
  cost,
  paidDiamonds,
  intrinsicValue,
  purchases
}
```

状态转移时再计算：

```text
beforePaid = paidDiamondsForPrice(state.spent)
afterPaid = beforePaid + paidDiamondsForPrice(action.cost)
rechargeBonus = marginalFreeDiamonds(beforePaid, paidDiamondsForPrice(action.cost))
rechargeValue = rechargeBonus * freeDiamondScore
newValue = state.value + action.intrinsicValue + rechargeValue
```

这样同一批、同一礼包组合，在不同路径下会因为购买前累计付费钻不同而得到不同的实际充值价值。

### 常驻钻石组合包补累充

规划器可以把非首次常驻钻石组合包作为买入批次内的“补累充”候选动作。补包规则为：

- 只使用 `src/constants/permanentPacks.json` 中名称为“钻石组合包 ...”且不包含“首次”的常驻包。
- 补包不触发限时组合包，不影响限时包升降档。
- 只有本批已经购买限时组合包时，才会枚举补包候选，不会生成“只买常驻包”的方案。
- 补包附着在该买入批次上，不会生成独立展示行。
- 补包只瞄准购买前的下一档每日累充，达到该档后立即停止，即使预算还有剩余也不会多买常驻包。
- 补包会消耗预算并参与后续动态规划剪枝，因此不同批次是否补包会影响后续可买路径。
- 页面按批次列出最终方案实际采用的常驻钻石组合包，例如 `第 1 批：钻石组合包 80 ×2`，不展示未被采用的补包候选。

补包选择使用无界背包。由于常驻钻石组合包满足：

```text
paidDiamonds = price / 2
```

同一补包总花费对应的每日累计付费钻相同，因此背包只需要在每个可达花费上保留内容价值最高的组合。之后再叠加该花费带来的边际累充赠钻：

```text
topUpValue =
  permanentOriginalValue(combo)
  + marginalFreeDiamonds(currentPaid, comboPaid) * freeDiamondScore
```

在剩余预算内，规划器先找到购买前的下一档每日累充，再选择能达到该档的最低花费组合；若花费相同，优先 `topUpValue` 更高、购买包数更少的组合。该补包候选会和“不补包”候选一起进入同一轮 DP 状态集合，由后续路径价值决定是否保留。

典型场景：

```text
限时包累计 5,900 付费钻
补 钻石组合包 80 ×2
累计变为 6,060 付费钻
解锁 6,000 档每日累充赠钻
```

### 排序与展示

方案排序仍为：

```text
总价值高 -> 购买数多 -> 花费少
```

页面展示区分两个口径：

- 礼包查询页展示单包 `RechargeValue`，属于 `1.2` 倍估算。
- 购买规划页展示实际累计赠钻、每批实际充值补正、补包列表和补包解锁的累充档位。
- “价值最优”“次优方案”“只买小包”“冲最大包”共用同一套边际赠钻和补包函数，保证不同方案价值可比较。

## 当前限制

- 批内买多个礼包时，只枚举价值排序后的前缀组合，不枚举所有子集。
- 剪枝会限制每个档位保留的候选数，因此极端输入下是受控近似搜索，不是无限状态的完整穷举。
- 主线进度解析由 `mainQuestProgress.js` 统一处理。
- 算法默认同一全局批次内的所有触发包共用一次升降档结算。
- 当前购买规划按单个每日累充周期计算，不模拟跨 0 点后的累充重置。

## 属性塔拓扑模式

在乐观卡包假设下，属性塔不再按蓝、红、翠、黄四个来源的第 N 批直接合并，而是先生成统一的属性塔事件流。

事件流由两类事件组成：

```text
全属性塔抵达事件：origin_group_all_towers
单属性塔触发事件：origin_tower_blue / red / green / yellow
```

二者按触发层排序：

```text
225 全属性塔抵达
250 单属性塔触发
275 全属性塔抵达
300 单属性塔触发
...
```

同一单属性塔层级下，已启用的蓝、红、翠、黄塔可以合并成同一属性塔批次；全属性塔抵达事件单独形成一个属性塔批次。

之后，整个属性塔事件流作为一个来源参与全局批次合并：

```text
全局第 N 批 = 主线第 N 批 + 无穷塔第 N 批 + 属性塔拓扑第 N 批 + ...
```

这样仍保留乐观卡包的同步能力，但不会出现全属性塔 275 被排到单属性塔 300 之后之类的拓扑倒置。

由于当前数据中属性塔相邻事件至少相隔 25 层，而跨日最多推进 20 层，跨日卡包不会改变相邻事件是否可同批。因此实现中不模拟真实日期，也不单独处理跨日窗口。

### 日期抽象原因

属性塔真实开放表为：

```text
周一：蓝塔
周二：红塔
周三：翠塔
周四：黄塔
周五：蓝塔 + 红塔
周六：翠塔 + 黄塔
周日：蓝塔 + 红塔 + 翠塔 + 黄塔
```

每个单属性塔每天最多推进 10 层。换日前后如果仍在同一个 2 小时限时包窗口内，单塔同批最多推进 20 层。

实现中没有把这些规则建成真实日期轴，而是抽象为属性塔拓扑序，原因是当前礼包数据满足：

```text
全属性塔抵达与相邻单属性塔触发相隔 25 层
单属性塔触发与相邻全属性塔抵达相隔 25 层
同类属性塔事件相隔 50 层
```

因此：

```text
跨日最大 20 层 < 相邻事件最小 25 层
```

跨日能力不会改变任何两个相邻属性塔礼包事件是否能落入同一批次，也不会改变事件先后顺序。算法只需要保证：

1. 全属性塔抵达按 `min(蓝塔, 红塔, 翠塔, 黄塔)` 触发。
2. 全属性塔抵达与单属性塔触发按层级排序。
3. 同一单属性层级下，已启用的蓝/红/翠/黄可在乐观卡包假设下同批。
4. 属性塔单批触发数固定为 1。

若后续礼包数据出现 20 层以内的属性塔事件间隔，则该抽象不再成立，需要改为开放窗口模型。
