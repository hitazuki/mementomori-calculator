# 限时组合包路径计算算法

本文记录购买方案规划器的路径计算算法，对应实现：

- `src/engine/ultraSalePlanner.js`
- `src/views/PackCalculatorView.vue`
- `test/ultraSalePlanner.test.js`

规则假设见 `planning-rules.md`。

流程图和伪代码速览见 `path-planning-visual.md`。

## 目标

在用户给定的预算（日元）、当前礼包档位、各触发源当前进度与计划推进范围内，计算一条或多条购买路径。

核心目标是：

```text
在总花费 <= 预算的约束下，最大化购买礼包的总评分价值
```

评分价值来自礼包 CE 计算结果，继续复用 `items-calc-formula.md` 与 `packCalc.js` 的物品评分逻辑。

当两条路径总价值相同时，排序规则为：

1. 购买礼包数量更多者优先。
2. 若购买数量也相同，花费更少者优先。

## 数据建模

### 礼包档位

算法从礼包数据中提取可用价格档位并升序排列，例如：

```text
160, 650, 1000, 1500, 3000, 6000, 11800
```

状态中不直接保存价格，而是保存 `tierIndex`，即当前处于第几个档位。

购买动作会让 `tierIndex + 1`，不购买动作会让 `tierIndex - 1`。档位会被限制在最低档和最高档之间。

### 触发机会

每个触发点会被整理成一个 `opportunity`：

```text
触发源 + 触发进度 + 各价格档位可买的礼包
```

例如主线 `10-28`、无穷塔 `180`、蓝塔 `80` 都会成为独立触发机会。

同一个触发点可能有多个价格档位的礼包。规划到某一批时，只会根据当时的 `tierIndex` 取对应档位的礼包。

### 触发进度排序

不同来源的触发进度会转换成可比较的数字：

- 普通数字触发点直接使用数字。
- 主线 `章节-关卡` 转成 `章节 + 关卡 / 100`，例如 `10-28` 转成 `10.28`。

每个触发源只保留：

```text
当前进度 < 触发点 <= 计划推到
```

范围内的触发机会。

## 批次生成

每个触发源先独立生成批次。

`2小时内最多触发` 表示该来源在一个卡包批次内最多放入多少个触发机会。例如无穷塔从 100 推到 130，若每批为 3，则会按排序后的触发机会每 3 个切成一批。

不同触发源相互独立，但可以同时触发。因此全局批次按批次序号合并：

```text
全局第 1 批 = 主线第 1 批 + 等级第 1 批 + 无穷塔第 1 批 + 各属性塔第 1 批
全局第 2 批 = 主线第 2 批 + 等级第 2 批 + 无穷塔第 2 批 + 各属性塔第 2 批
...
```

每个全局批次统一应用一次升降档规则：

- 批内至少买 1 个礼包：下一批升 1 档。
- 批内全不买：下一批降 1 档。

## 全属性塔抵达

“全属性塔抵达”不是用户单独填写的触发源，而是从四个属性塔推导。

算法要求蓝塔、红塔、翠塔、黄塔四个来源都启用，才会尝试派生全属性塔礼包。

对于某个全属性塔触发层 `floor`：

```text
四塔共同进度 = min(蓝塔层数, 红塔层数, 翠塔层数, 黄塔层数)
```

只有四塔在规划范围内都能达到该层时，该触发机会才有效。

派生触发机会所属批次取四塔中最晚到达该层的批次：

```text
派生批次 = max(蓝塔到达批次, 红塔到达批次, 翠塔到达批次, 黄塔到达批次)
```

然后把该机会加入对应的全局批次。

## 动态规划

最优路径使用动态规划计算。

### 状态

每个状态表示处理完若干批次后的路径结果：

```js
{
  tierIndex,   // 下一批开始时的礼包档位
  spent,       // 已花费日元
  value,       // 已获得礼包总评分价值
  purchases,   // 已购买礼包数量
  steps        // 路径明细
}
```

状态表 key 为：

```text
tierIndex | spent
```

同一档位、同一花费下，可能保留少量不同路径，用于生成次优方案。

### 动作

对每个全局批次，根据当前档位找到批内可买礼包。

批内动作包括：

1. 全不买。
2. 买 1 个。
3. 买 2 个。
4. 以此类推，最多买完整个批次内当前档位存在的礼包。

批内可买礼包会先按评分价值从高到低排序。买多个时，算法只考虑“从高价值开始买前 N 个”的前缀动作。

这样可以避免枚举批内所有子集。因为在同一档位、同一价格下，预算成本随购买数量线性增加，买 N 个时选择价值最高的 N 个不会差于其他组合。

### 转移

对每个状态和每个动作，生成候选新状态：

```text
newSpent = spent + action.cost
```

若 `newSpent > budget`，丢弃该候选。

否则根据是否购买更新档位：

```text
action.bought ? tierIndex + 1 : tierIndex - 1
```

并累加价值、购买数量和路径明细。

### 剪枝

触发源很多时，状态数会快速膨胀。当前实现每处理完一个批次后执行剪枝。

剪枝分两步：

1. 按 `tierIndex` 分组。
2. 在每个档位内按花费从低到高扫描，只保留 Pareto 状态。

Pareto 状态含义：

```text
若某状态花费更高，价值却不高于已有状态，则它不可能成为更优路径，可以丢弃。
```

之后再按路径排序规则保留前 `maxStatesPerTier` 个状态。页面当前使用 `350`，引擎默认值也是 `350`，并设置最低有效值 `50`。

这是一种近似受控的动态规划：保留足够多的高价值候选，同时避免 7 个触发源全启用时浏览器卡死。

## 多方案生成

页面展示的方案分两类。

### 价值最优 / 次优方案

这两个方案来自同一次动态规划。

动态规划结束后，把所有保留状态按排序规则排列：

```text
总价值高 -> 购买数多 -> 花费少
```

取第 1 条作为“价值最优”，取第 2 条作为“次优方案”。

为了避免同一条路径重复出现，算法会用路径签名去重。签名由每批档位与购买触发点组成。

### 策略方案

策略方案不用动态规划搜索，而是按固定规则模拟同一批次列表。

“只买小包”：

- 只有当前档位是最低档时才买。
- 买入当前批内价值最高的 1 个最低档礼包。
- 升档后等待超时掉回最低档。

“冲最大包”：

- 未到最高档前，每批买 1 个价值最高礼包用于升档。
- 到最高档后，在预算内尽量购买当前批内的高档礼包。

策略方案与最优方案共享同一套触发批次、预算限制和升降档规则，只是动作选择方式不同。

## 路径展示压缩

计算结果中的每个 `step` 对应一个全局批次。

为了让页面更容易阅读，展示层会压缩连续不购买的批次：

```text
不买第 2 批 + 不买第 3 批 + 不买第 4 批
```

显示为：

```text
第 2-4 批，连续不买 ×3
```

购买批次不会合并，因为用户需要展开查看每个礼包的触发点和内容物。

压缩逻辑由 `compressUltraSalePlanSteps()` 处理，并有单元测试覆盖。

## 复杂度

设：

- `B` 为全局批次数。
- `S` 为剪枝后每批保留状态数。
- `A` 为单批可选动作数。
- `T` 为档位数。

动态规划主体近似为：

```text
O(B * S * A)
```

剪枝会按档位处理状态，实际状态上限受 `maxStatesPerTier * T` 控制。

单批动作数通常等于：

```text
1 + 当前批内当前档位可买礼包数
```

其中 `1` 是“全不买”动作。

## 当前限制

- 批内买多个礼包时，只枚举价值排序后的前缀组合，不枚举所有子集。
- 剪枝会限制每个档位保留的候选数，因此极端输入下是受控近似搜索，不是无限状态的完整穷举。
- 主线进度用 `章节 + 关卡 / 100` 排序，适用于当前章节-关卡格式。
- 算法默认同一全局批次内的所有触发包共用一次升降档结算。
