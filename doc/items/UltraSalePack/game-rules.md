# 超值限时组合包游戏内规则

本文档只记录游戏内已经确认的规则、数据关系和时间边界。规则文档确定后一般不随实现调整改动；如果游戏数据或规则本身变化，再更新本文档。

购买规划器的需求见 `requirements.md`，当前实现设计见 `implementation-design.md`。

## 档位

超值限时组合包的档位由付费钻数量决定。项目数据用日元价格表示，同一档位的对应关系如下：

| 价格 | 付费钻 |
| ---: | ---: |
| 160 | 80 |
| 650 | 325 |
| 1000 | 500 |
| 1500 | 750 |
| 3000 | 1500 |
| 6000 | 3000 |
| 11800 | 5900 |

初始触发的限时包通常是 80 付费钻档，也就是价格 160 档。

## 升降档

- 触发并购买至少 1 个限时包后，下个结算批次升 1 档。
- 触发但没有购买，2 小时后超时，下个结算批次降 1 档。
- 档位不会低于最低档，也不会高于最高档。

## 卡包批次

当 2 小时内连续触发多个限时包时，这些礼包视作同一批：

- 在最后一个礼包消失前，批内至少购买 1 个，则下一批升 1 档。
- 批内其他礼包不买，不会额外导致降档。
- 批内全部不买，则下一批降 1 档。

## 触发源

限时组合包可以由不同来源触发。当前规划相关来源包括：

- 主线
- 玩家等级
- 无穷塔
- 蓝塔
- 红塔
- 翠塔
- 黄塔
- 全属性塔抵达

不同类别的限时包可以同时触发。

主线和无穷塔可以随推关连续触发。单个属性塔每天只能爬 10 层，不能像无穷塔一样持续触发。玩家等级提升较慢，通常不会连续大量触发。

## 全属性塔抵达

“全属性塔抵达”由蓝塔、红塔、翠塔、黄塔共同决定：

```text
全属性塔进度 = min(蓝塔层数, 红塔层数, 翠塔层数, 黄塔层数)
```

只有四个属性塔都达到某个“全属性塔抵达”礼包层级时，该礼包才会触发。

## 属性塔开放

属性塔存在真实日期限制：

| 星期 | 开放属性塔 |
| --- | --- |
| 周一 | 蓝塔 |
| 周二 | 红塔 |
| 周三 | 翠塔 |
| 周四 | 黄塔 |
| 周五 | 蓝塔、红塔 |
| 周六 | 翠塔、黄塔 |
| 周日 | 蓝塔、红塔、翠塔、黄塔 |

属性塔按每天 4 点换日。每个单属性塔每天最多推进 10 层。

若玩家在 4 点换日前后 2 小时内卡包，单个属性塔在同一个限时包批次内理论上最多可以跨两天推进：

```text
跨日同批最大推进 = 10 + 10 = 20 层
```

## 属性塔事件间隔

当前礼包数据中，属性塔相关礼包的相邻触发事件至少相隔 25 层：

- 全属性塔抵达与下一个单属性塔触发相隔 25 层。
- 单属性塔触发与下一个全属性塔抵达相隔 25 层。
- 同类事件相隔 50 层。

因此在当前数据下，即使考虑 4 点换日前后 2 小时卡包，单塔最多推进 20 层，仍无法跨过相邻属性塔礼包事件。

如果未来属性塔或全属性塔礼包触发间隔缩短到 20 层以内，需要重新评估规划器是否必须引入真实日期和开放窗口模型。

## 每日累充

每日累充按 0 点重置，属性塔按 4 点换日。这是两套不同时间规则：

```text
累充重置：00:00
属性塔换日：04:00
```

每日累充赠钻规则记录在 `../items-calc-formula.md`。

## 常驻钻石组合包

常驻钻石组合包数据来自 `src/constants/permanentPacks.json`。购买常驻钻石组合包会增加每日累计付费钻，但不会触发超值限时组合包，也不会影响限时包升降档。
