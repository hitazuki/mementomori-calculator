# 超值限时组合包购买决策规划需求

本文档记录购买规划器需要满足的产品行为和用户可见规则。游戏内稳定规则见 `game-rules.md`；算法、性能策略和实现取舍写入 `implementation-design.md`。

本文只描述规划器如何基于游戏规则、玩家预期性价比、机会偏好和计算前瞻范围，判断当前哪些礼包值得买、哪些机会值得触发、哪些机会应保留。

---

## 1. 需求定位

购买规划器不是预算花费器，也不是触发机会消耗器。

玩家不是给规划器一笔钱让它花完，而是给规划器一个“什么样的购买值得”的标准；玩家也不是给规划器一堆机会让它用完，而是给规划器一个当前起点，让规划器判断哪些机会现在值得用、哪些机会应留到未来。

核心目标：

```text
基于预期性价比、机会偏好和当前触发起点，
在计算前瞻范围内生成购买、触发、跳过和保留决策。
```

规划器允许输出“暂不购买 / 暂不触发 / 保留机会”。如果当前没有达到玩家标准的路径，空方案是有效方案。

---

## 2. 核心概念

### 2.1 预期性价比

预期性价比表示玩家对“花钱是否值得”的最低要求。

建议单位：

```text
性价比比值 (CE倍率)
```

记：

```text
expectedRatio = 玩家预期性价比
```

与项目统一的性价比计算方法保持一致：`CE = FinalValue / PaidDiamonds`，其中 `PaidDiamonds = 礼包价格(日元) / 2`。
例如 `expectedRatio = 20` 表示玩家期望礼包的实际总价值至少是其“付费钻石基准”的 20 倍。

降低预期性价比表示玩家愿意牺牲部分金钱效率，换取当前机会被利用、路径推进、升档或累充收益兑现；这不等同于“更珍惜机会”。如果玩家想表达“不想轻易消耗触发机会”，应调整机会偏好。

### 2.2 机会偏好

机会偏好表示玩家对“触发机会是否值得现在用”的倾向。

页面可提供三档：

| 机会偏好 | 含义 | 行为倾向 |
| --- | --- | --- |
| 积极使用机会 | 当前路径达标即可使用机会 | 更容易触发并购买 |
| 均衡 | 比较当前收益与保留价值 | 默认 |
| 珍惜机会 | 除非当前明显更优，否则保留机会 | 更少消耗稀缺触发机会 |

机会偏好不直接替代性价比。两者分别回答：

```text
预期性价比：这笔钱花得值不值？
机会偏好：这个触发机会现在用值不值？
```

### 2.3 计算前瞻范围

计算前瞻范围表示规划器本次向后纳入比较的触发机会范围，用于判断当前机会的机会成本、保留价值和后续路径收益。

计算前瞻范围参与计算，但不是玩家必须推进到的位置，也不是最终必须展示的范围。

页面应区分：

| 字段 | 含义 | 是否参与计算 |
| --- | --- | --- |
| 计算前瞻范围 | 纳入路径搜索与机会成本比较的未来范围 | 是 |
| 展示范围 / 展示明细深度 | 页面展示多少批次、节点和明细 | 否 |

---

## 3. 输入

页面字段：

- 预期性价比：玩家认为值得购买的最低性价比，与计算器统一 CE 单位一致（如 20 倍）。
- 机会偏好：积极使用机会 / 均衡 / 珍惜机会。
- 当前礼包档位：玩家当前已处于的限时组合包档位。
- 各触发源当前进度：玩家当前已达到的位置；规划器只考虑大于该值的触发点。
- 起始触发机会：从当前进度出发，下一批可能被安排触发的机会集合；可由当前进度和礼包触发表自动推导，也可在高级模式下手动指定。
- 计算前瞻范围：规划器向后纳入比较的触发机会范围。
- 展示范围 / 展示明细深度：只影响页面展示，不影响计算。
- 2 小时内最多触发：同一触发源在同一个卡包批次内最多能连续触发多少个限时包；这是同批上限，不要求必须合并到上限。
- 可选安全上限：玩家不希望本次建议超过的总花费。该字段只作为风险过滤条件，不作为必须花完的预算目标。
- 补累充模式：关闭 / 自动价值判断 / 手动确认。

输入格式：

- 主线进度支持 `章节-关卡`，例如 `13-28`；也支持游戏内连续关卡进度，例如 `336`。
- 其他来源使用数字层数或等级。
- “全属性塔抵达”由四个属性塔自动派生，页面不单独提供起始层、目标层或每批触发数。

字段替换关系：

| 旧字段 | 新字段 | 调整原因 |
| --- | --- | --- |
| 主预算 | 预期性价比 + 可选安全上限 | 玩家给的是值得买标准，不是必须花完的钱 |
| 补累充阈值 | 补累充模式 + 价值判断 | 补累充也是购买行为，应进入性价比评价 |
| 计划推到 | 计算前瞻范围 | 表示计算视野，不是推进目标 |
| 当前触发机会数量 | 起始触发机会 | 表示当前规划起点，不表示必须消耗 |
| 价值最优 | 决策价值最高 | 同时考虑钱、机会、执行成本和未来状态 |

## 4. 路径价值模型

### 4.1 路径花费与价值

为保持核心规划的纯净度，规划器主路径的花费与价值**仅包含限时组合包**：

```text
路径总花费 = 限时组合包花费
```

```text
路径总价值 = 限时组合包内容 CE + 实际每日累充新增赠钻 CE
```

*注意：自动补累充已被彻底解耦为展示层的附加建议（见第 9 节）。如果系统最终建议玩家补包，补包的花费和额外价值会作为独立的 Tip 附加展示，不再强行揉入主路径花费中，以避免污染玩家的安全预算上限。*

### 4.2 金钱净收益

```text
moneySurplus =
路径总价值 - (路径总花费 / 2) × expectedRatio
```

`moneySurplus >= 0` 表示路径在金钱效率上达到玩家预期。

金钱净收益仍是核心指标，但不能作为唯一排序标准，因为它没有完整体现触发机会的保留价值。

### 4.3 机会状态价值

规划器应评价“动作后的状态”，而不是只评价“已买路径”。

```text
stateValue =
从某个状态继续规划时，在计算前瞻范围内可获得的最佳决策价值
```

状态至少包括：

- 当前礼包档位。
- 各触发源前沿。
- 已触发 / 已购买 / 已保留节点。
- 当前累充日累计付费钻。
- 当前累充日已推进过的属性塔集合。
- 剩余可达触发机会。
- 是否可跨日重置。
- 机会偏好。

机会成本通过反事实比较体现：

```text
机会成本 =
保留该机会后的最佳状态价值
- 消耗该机会后的最佳状态价值
```

如果保留机会未来更优，则当前消耗机会的成本高；如果机会没有更好用法，则成本低。

### 4.4 决策价值

规划器以路径决策价值作为主排序指标。

```text
decisionValue =
moneySurplus
+ remainingStateValue
- executionCost
- riskCost
```

也可以用搜索 / 动态规划形式表达：

```text
推荐动作 =
argmax(当前动作收益 + 后续状态最佳价值)
```

其中：

- `moneySurplus` 判断钱是否花得值。
- `remainingStateValue` 判断机会是否值得现在用。
- `executionCost` 表示同批触发压力、批次数、触发但不买、跨日复杂度等执行成本。
- `riskCost` 表示补累充、较高总花费、复杂操作等风险成本。

排序优先级：

1. 决策价值最高。
2. 金钱净收益更高。
3. 关键机会消耗更少。
4. 实际性价比更高。
5. 执行压力更低。
6. 补累充风险更低。
7. 总花费更低。

## 5. 触发机会与动作

规划器处理的触发源包括：

- 主线
- 玩家等级
- 无穷塔
- 蓝塔 / 红塔 / 翠塔 / 黄塔
- 全属性塔抵达

每个触发机会可处于以下状态：

- 可达：在当前前沿、计算前瞻范围和拓扑约束下，理论上可安排触发。
- 本批触发：被选入当前候选批次，视为已触发，并推进对应来源前沿。
- 已购买：本批触发节点中实际购买的限时包。
- 触发未买：本批触发但不购买，批内全不买时下一批降档。
- 保留：当前可达但未触发，不推进来源前沿，可留到后续批次或未来规划。

可选动作：

- 保留机会。
- 触发并购买。
- 触发但不购买，用于降档或跳过低价值档位。
- 同批部分购买。
- 合批或拆批。
- 跨日重置累充。
- 自动补累充或手动确认补累充。

默认原则：

```text
如果当前触发机会不能带来高于保留机会的决策价值，
应优先保留，而不是为了消耗机会而触发。
```

## 6. 拓扑约束

规划器采用玩家主动控制触发时机的“乐观卡包”假设：玩家可以控制主线、无穷塔、属性塔等来源的推进节奏，在限定范围内把触发机会安排到一个或多个 2 小时卡包批次中。

乐观卡包不代表可以任意打乱触发顺序。

### 6.1 普通触发源

同一触发源内必须保持顺序，不能把后续触发点排到前置触发点之前。

### 6.2 属性塔

蓝塔、红塔、翠塔、黄塔各自维护独立前沿。某个单塔节点被选入本批触发节点后，只推进该塔自己的前沿；未选入当前批次的同层单塔节点仍可保留。

同层不同单塔之间没有直接先后关系。可达节点可以合批，也可以拆批；批次编排只需满足：

- 同一来源自身顺序。
- 同批触发上限。
- 全属性塔抵达派生门槛。

### 6.3 全属性塔抵达

全属性塔抵达由四塔进度共同派生：

```text
全属性塔进度 = min(蓝塔层数, 红塔层数, 翠塔层数, 黄塔层数)
```

只有四塔都达到某个全属性塔抵达礼包层级时，该节点才可触发。

在规划器前沿模型中，判定方式为检查每个单塔前沿区间是否覆盖该 allTower 层：

```text
previous < allTowerLayer <= next
```

任一塔的前沿区间未覆盖该层时，该全属性塔抵达节点不可达。

全属性塔抵达节点本身不推进蓝塔、红塔、翠塔或黄塔的单塔前沿；它只是由四塔共同派生出的触发机会。

触发全属性塔抵达时，四塔中“最后达到该层”的塔不能在同一批次中同时触发其下一单塔节点。该塔刚跨过全属性塔门槛，尚未越过该门槛后的单塔触发栅栏。

## 7. 批次与升降档

一个全局结算批次表示一个 2 小时卡包窗口。不同触发源若能通过控进度安排到同一 2 小时窗口，可以进入同一个全局结算批次。

合批与拆批规则：

- 可同批不代表必须同批。
- 可拆批不代表必须拆批。
- 同一来源在同批内触发次数不能超过“2 小时内最多触发”。
- 合批不按固定来源批次序号决定，只要满足来源顺序、同批上限和属性塔 / 全属性塔约束即可。
- 同批触发数越多，执行压力越大；在决策价值相近时，优先选择触发数更少、批次压力更低的方案。

升降档规则：

- 批内至少购买 1 个限时包，则下一批升档。
- 批内全不买，则下一批降档。
- 批内部分购买时，只要至少购买 1 个，批内未购买礼包不会额外导致降档。
- 档位不低于最低档，不高于最高档。

批次编排会影响：

- 升降档次数。
- 后续可购买档位。
- 同一累充日内累计付费钻。
- 是否跨 0 点重置累充。
- 是否可补累充。
- 触发机会被消耗还是保留。

因此批次编排属于购买决策的一部分，不是单纯展示分组。

## 8. 累充日模型

规划器不要求用户填写真实日期、开始时间或规划天数，使用抽象累充日处理：

- 批次顺序。
- 2 小时超时。
- 0 点累充重置。
- 属性塔推进带来的强制累充日约束。

不同批次可以共用同一累充日进度，也可以在批次之间跨 0 点并重置每日累充进度。每日累充重置只在两个全局批次之间发生；同一全局批次内的限时包购买和补累充购买视为发生在同一累充日内。

### 8.1 属性塔强制重置

记：

```text
A = 当前累充日内已经推进过的单属性塔集合
B = 当前候选批次中被选为本批触发节点的单属性塔集合
```

判定规则：

- 若 `A ∩ B` 非空，则当前候选批次执行前必须先跨日重置累充。
- 若 `A ∩ B` 为空，则当前候选批次可以继续沿用当前累充日，并在批次执行后将 `B` 合并进 `A`。
- 首个批次 `A` 为空，不会因为属性塔推进而强制预先重置。
- 可达但未进入当前候选批次的属性塔节点，不参与 `B`。
- 不同属性塔在同一累充日内推进不强制重置。
- 同一属性塔在同一累充日内再次推进才强制重置。
- 全属性塔抵达节点本身不推进四个单塔前沿，因此不计入 `B`。

当前乐观卡包策略不使用跨 4 点额外推塔能力。若未来新增精确跨 4 点属性塔模式，则同一个 2 小时批次不能同时作为跨 0 点累充批次和跨 4 点属性塔批次。

### 8.2 每日累充赠钻

购买规划页按路径内同一累充日累计付费钻计算实际赠钻：

```text
本次赠钻 =
TotalBonusFreeDiamonds(购买后累计付费钻)
- TotalBonusFreeDiamonds(购买前累计付费钻)
```

不同批次共用同一累充日时，累计付费钻持续累加；跨日重置后，新的累充日从 0 开始计算。

每日累充赠钻 CE 必须计入路径总价值。

## 9. 自动补累充的彻底解耦 (UI Post-Processing Top-up)

补累充不再作为核心路径规划的内置动作，而是彻底降级为展示层的独立**后处理模块**。这保证了规划器不强迫玩家购买常驻包，仅提供基于数学证明的“绝对不亏超值提醒”。

### 9.1 价值判定与触发机制

当规划器生成最终的限时包候选方案后，展示层会对方案中每一天的累计付费钻进行独立扫描：
```text
topUpSurplus = 补包新增累充赠钻CE + 补包内容CE - (补包花费 / 2) * expectedRatio
```

*   如果 `topUpSurplus >= 0`：在对应批次下方追加一条非强制性的高亮【💡补包建议 Tip】（例如：建议补充 ¥160 常驻包以激活下一档累充）。
*   如果 `topUpSurplus < 0`：无事发生，不展示。

### 9.2 解耦的优势与规则

- **不拉低主方案收益**：由于 `topUpSurplus >= 0` 保证了这笔附加投资自身的性价比已经达到玩家期望，把它附加上去只会提升整体收益，绝对不会拉低原路线的综合性价比。
- **不占用安全上限**：补包推荐的花费**独立展示**，不计入 DP 的 `totalSpentYen`，不会因为被系统“硬塞常驻包”导致触发安全上限而意外淘汰优秀主方案。
- **补包规则**：采用能达到目标累充档的最低花费组合。若花费相同，优先内容价值更高、包数更少的常驻组合包。

## 10. 候选路径生成与停止

规划器从起始触发机会开始生成候选路径。每一步可以选择：

- 保留当前可达机会。
- 将一个或多个可达机会安排进当前批次。
- 当前批次购买全部、购买部分或全部不买。
- 当前批次后继续沿用同一累充日。
- 当前批次后跨 0 点重置累充日。
- 跨日前按补累充模式处理补累充。
- 等待 2 小时使批次超时并触发降档。

路径必须满足：

- 来源内顺序。
- 同批触发上限。
- 属性塔 / 全属性塔抵达约束。
- 升降档规则。
- 累充日规则。
- 计算前瞻范围。
- 可选安全上限。
- 补累充安全设置。

停止扩展条件：

- 后续所有动作的决策价值均无法改善。
- 继续扩展只会降低综合性价比，且无法通过升档、跨日、补累充或保留机会恢复。
- 已达到计算前瞻范围。
- 已达到可选安全上限。
- 后续没有可达触发机会。

停止表示继续买或继续触发不符合当前偏好，不是规划失败。

## 11. 方案类型

页面同时生成多种方案，供玩家按偏好切换。

### 11.1 推荐方案

默认主方案。目标是在玩家预期性价比和机会偏好下，选择决策价值最高的方案。

可包含升档铺路、触发不买降档、拆批、跨日和补累充，但必须解释这些动作为什么提高决策价值。

### 11.2 保守方案

只推荐确定达到预期性价比且机会消耗较少的动作。

特点：

- 少买铺路包。
- 少触发后不买。
- 更倾向保留机会。
- 实际性价比通常更高，但总收益可能低于推荐方案。

### 11.3 升档铺路方案

允许购买若干低档礼包作为升档成本，以换取后续更高档礼包收益。

要求：

```text
包含铺路成本后的整条路径实际性价比 >= 玩家预期性价比
且决策价值高于不铺路方案
```

### 11.4 降档等待方案

允许触发但不购买某些低价值批次，通过 2 小时超时降档，使后续礼包回到更适合购买的档位。

必须展示被消耗的触发机会，并说明为什么消耗这些机会比直接保留更优。

### 11.5 机会保留方案

当当前起点之后没有达到预期的路径，或当前使用机会不如保留机会时，生成机会保留方案。

展示：

- 当前不建议购买。
- 当前不建议继续触发的节点。
- 保留原因。
- 若降低预期性价比到多少，或调整机会偏好到什么程度，才会出现可推荐路径。
- 哪些未来状态可能让这些机会重新值得使用。

## 12. 展示

### 12.1 汇总

规划结果汇总应显示：

- 推荐结论：建议购买 / 建议暂不购买 / 建议先降档 / 建议保留机会。
- 预期性价比。
- 机会偏好。
- 路径实际性价比。
- 金钱净收益。
- 决策价值。
- 总价值。
- 总花费。
- 限时包花费。
- 补累充花费。
- 购买限时包数。
- 已触发机会数。
- 已购买机会数。
- 保留机会数。
- 跨日重置次数。
- 补累充批次数。
- 下一批结束后的礼包档位。

若存在可选安全上限，应显示当前方案是否触达上限，以及因安全上限被过滤的高价值方案数量。

### 12.2 批次列表

列表层展示：

- 批次编号。
- 本批动作：购买 / 不买 / 部分购买 / 保留。
- 本批触发节点数。
- 本批购买节点数。
- 本批限时包花费。
- 本批补累充花费。
- 本批新增价值。
- 本批金钱净收益。
- 本批后累计实际性价比。
- 是否跨日重置。
- 是否补累充。
- 本批后下一档位。
- 本批决策理由。

连续不购买的批次可以压缩展示：

```text
第 2-4 批，连续触发不买 ×3，用于降档
```

但压缩展示必须保留展开明细，避免玩家误解为机会没有被消耗。

### 12.3 展开明细

展开明细显示：

- 每个礼包的触发源与触发点。
- 触发前档位。
- 购买价格与付费钻。
- 礼包内容 CE。
- 每日累充新增赠钻 CE。
- 补累充新增价值。
- 单包性价比。
- 路径级增量性价比。
- 是否购买。
- 未购买原因。
- 是否作为升档铺路。
- 是否作为降档等待。
- 是否建议保留。
- 补累充使用的常驻钻石组合包。

### 12.4 保留机会展示

保留机会必须显式展示：

- 触发源。
- 触发点。
- 当前不建议触发的原因。
- 保留后可能等待的条件：档位变化、累充日重置、后续更高价值机会、预期性价比降低、机会偏好调整等。
- 是否属于关键机会。

## 13. 计算触发

规划器不在输入变化时自动重算，需要用户点击“计算购买方案”。

如果用户只调整预期性价比、机会偏好或展示范围，页面可以复用上一次候选路径集合重新排序和过滤。

如果用户调整当前进度、起始触发机会、计算前瞻范围、同批触发上限、补累充模式或安全上限，则必须重新生成或重新校验候选路径。

## 14. 验收标准

### 14.1 不再花完预算

没有主预算花完目标。若当前路径低于预期性价比，规划器应输出暂不购买或保留机会。

### 14.2 不再用完机会

计算前瞻范围内存在大量机会时，规划器可以只建议使用其中一小部分，并明确展示保留机会和原因。

### 14.3 净收益不是唯一排序

净收益仍用于判断钱是否花得值，但主排序必须使用包含机会状态价值和执行成本的决策价值。

### 14.4 支持铺路但必须路径级达标

低档包可作为升档铺路，但包含铺路成本后的整条路径必须达到预期性价比，并且决策价值高于不铺路。

### 14.5 支持降档但必须解释机会损耗

若推荐触发不买以降档，必须展示被消耗的触发机会、机会成本和增量收益。

### 14.6 补累充逻辑彻底解耦

补累充不再计入主路径的总花费，它被彻底移出 DP 核心逻辑。补包策略仅作为 UI 层的独立附加功能：当满足 `topUpSurplus >= 0` 时，以独立的非强制性 Tip 形式展示给玩家，绝不会因搭售常驻包而拉低玩家体验。

### 14.7 前瞻范围参与计算但不是目标

计算前瞻范围用于判断机会成本和未来状态价值，不表示玩家必须推进到该范围终点。

### 14.8 空方案是有效方案

如果所有候选路径都低于预期，或当前使用机会不如保留机会，规划器必须输出有效空方案：

```text
当前不建议购买；
当前不建议继续触发；
建议保留以下机会；
若预期性价比降至 X 或机会偏好调整为 Y，才会出现可推荐路径。
```
