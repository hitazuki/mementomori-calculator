# 超值限时组合包购买规划需求

本文档记录购买规划器需要满足的产品行为和用户可见规则。游戏内稳定规则见 `game-rules.md`；算法、性能策略和实现取舍写入 `implementation-design.md`。

本文只描述规划器如何使用游戏规则生成购买路径，不重复展开游戏规则本身。

## 1. 目标

购买规划器在用户给定主预算、补累充阈值、当前礼包档位、各触发源当前进度与计划推进范围后，生成一条或多条购买路径。

核心目标：

```text
在限时包花费 <= 主预算的约束下，
最大化购买路径总价值。
补累充花费由阈值判定自动确定，不计入预算约束，仅在结果中展示。
```

要求：

- 礼包内容价值继续使用 `../items-calc-formula.md` 的 CE 评分逻辑。
- 购买规划页的每日累充赠钻按路径内实际累计付费钻计算，不使用单包 `1.2` 倍估算。
- 同批触发礼包越多，玩家执行压力越大；在不降低总价值、预算可行性和升降档可达性的前提下，优先选择同批触发数更少、批次压力更低的方案。
- 可同批触发的机会允许拆到不同批次；跨累充日重置后获取更多每日累充奖励的路径也应纳入候选。
- 若主预算未用完，应是因为继续购买限时包会超预算，或后续在当前档位没有可购买礼包。补累充花费不参与此判断。

## 2. 输入

页面字段：

- 主预算（日元）：本次可用于购买限时组合包的预算。补累充花费由阈值判定独立计算，不受主预算约束。
- 补累充阈值 x%：跨日前判断是否补包的临界百分比，默认 `10%`。含义见 §8。
- 当前礼包档位：玩家当前已经处于的限时组合包档位。
- 当前层 / 等级：触发源当前已达到的位置；规划器只考虑大于该值的触发点。
- 计划推到：该触发源本次规划最多推进到的位置。
- 2 小时内最多触发：同一触发源在同一个卡包批次内最多能连续触发多少个限时包；这是同批上限，不要求必须合并到上限。

输入格式：

- 主线进度支持 `章节-关卡`，例如 `13-28`；也支持游戏内连续关卡进度，例如 `336`。
- 其他来源使用数字层数或等级。
- 页面可以预设中等触发区间以降低首次试算压力，玩家仍可手动扩大“计划推到”范围。
- “全属性塔抵达”由四个属性塔自动派生，页面不单独提供起始层、目标层或每批触发数。

## 3. 触发机会

规划器处理的触发源包括：

- 主线
- 玩家等级
- 无穷塔
- 蓝塔 / 红塔 / 翠塔 / 黄塔
- 全属性塔抵达（由四个属性塔自动派生）

规划器需要区分三类状态：

- 可触发节点：在当前前沿、计划范围和拓扑约束下，理论上可被安排触发的节点。
- 本批触发节点：从可触发节点中选入当前候选批次的节点；这些节点视为已触发，并推进对应来源前沿。
- 已购买节点：本批触发节点中实际购买的限时包。

未进入当前候选批次的可触发节点不视为已触发，不推进来源前沿，可以保留到后续批次。因此规划器可以先通过其他触发节点升档，再在更高档位触发并购买保留节点对应的礼包。

## 4. 拓扑约束

规划器采用玩家主动控制触发时机的“乐观卡包”假设：玩家可以控制主线、无穷塔、属性塔等来源的推进节奏，在限定范围内把触发机会安排到一个或多个 2 小时卡包批次中。

乐观卡包不代表可以任意打乱触发顺序。触发节点必须满足以下拓扑约束。

### 4.1 普通触发源

同一触发源内必须保持顺序，不能把后续触发点排到前置触发点之前。

### 4.2 属性塔

属性塔按“每个单塔独立前沿 + 全属性塔派生门槛”建模：

```text
全属性塔抵达 225
各单属性塔 250
全属性塔抵达 275
各单属性塔 300
...
```

蓝塔、红塔、翠塔、黄塔各自维护独立前沿。某个单塔节点被选入本批触发节点后，只推进该塔自己的前沿；未选入当前批次的同层单塔节点仍可保留到后续批次。

同层不同单塔之间没有直接先后关系。可达节点可以合批，也可以拆批；批次编排只需要满足：

- 同一来源自身顺序。
- 同批触发上限。
- 全属性塔抵达的派生门槛。

### 4.3 全属性塔抵达

全属性塔抵达由四塔进度共同派生：

```text
全属性塔进度 = min(蓝塔层数, 红塔层数, 翠塔层数, 黄塔层数)
```

只有四塔都达到某个全属性塔抵达礼包层级时，该节点才可触发。其最早可触发位置由四塔中最慢达到该层的塔决定。

在规划器前沿模型中，判定方式为检查每个单塔前沿区间是否覆盖该 allTower 层：

```text
previous < allTowerLayer <= next
```

任一塔的前沿区间未覆盖该层时，该全属性塔抵达节点不可达。全属性塔抵达节点本身不推进蓝塔、红塔、翠塔或黄塔的单塔前沿；它只是由四塔共同派生出的触发机会。

触发全属性塔抵达时，四塔中“最后达到该层”的塔不能在同一批次中同时触发其下一单塔节点。该塔刚跨过全属性塔门槛，尚未越过该门槛后的单塔触发栅栏。

满足派生门槛后，全属性塔抵达机会可以与其他满足约束的触发机会合批，也可以拆到独立批次。

## 5. 批次编排

一个全局结算批次表示一个 2 小时卡包窗口。不同触发源若能通过控进度安排到同一 2 小时窗口，可以进入同一个全局结算批次。

合批与拆批规则：

- 可同批不代表必须同批。
- 可拆批不代表必须拆批。
- 同一来源在同批内触发次数不能超过“2 小时内最多触发”。
- 合批不按固定来源批次序号决定，只要满足来源顺序、同批上限和属性塔 / 全属性塔约束即可。

分批不是单纯的展示拆分。某个触发机会被安排到独立批次后，会改变后续批次开始时各触发源的已推进位置，并影响后续可生成、可合批的触发节点。

批次编排会影响：

- 升降档次数。
- 后续可购买档位。
- 同一累充日内的累计付费钻。
- 是否有机会跨 0 点重置每日累充进度。
- 是否能在跨日前使用常驻钻石组合包补累充。

因此批次编排本身属于购买路径选择的一部分，而不是单纯的展示分组。

## 6. 升降档

每个全局结算批次统一应用一次升降档规则：

- 批内至少购买 1 个限时包，则下一批升档。
- 批内全不买，则下一批降档。

升降档不低于最低档，也不高于最高档。具体档位、价格和升降档游戏规则以 `game-rules.md` 为准。

## 7. 累充日模型

规划器不要求用户填写真实日期、开始时间或规划天数。规划器使用抽象累充日处理：

- 批次顺序。
- 2 小时超时。
- 0 点累充重置。
- 属性塔推进带来的强制累充日约束。

不同批次可以共用同一个累充日进度，也可以在批次之间跨过 0 点并重置每日累充进度。是否跨日会影响每日累充奖励，因此应作为购买路径候选的一部分。

每日累充重置只在两个全局批次之间发生。同一个全局批次内的所有限时包购买和补累充购买都视为发生在同一个累充日内，不模拟批内跨 0 点购买顺序。

### 7.1 属性塔强制重置

属性塔相关的强制累充重置与拓扑序无关，而是由同一累充日内已经推进过的单属性塔集合决定。

记：

```text
A = 当前累充日内已经推进过的单属性塔集合
B = 当前候选批次中被选为本批触发节点的单属性塔集合
```

判定规则：

- 若 `A ∩ B` 非空，则当前候选批次执行前必须先跨日重置累充。
- 若 `A ∩ B` 为空，则当前候选批次可以继续沿用当前累充日，并在批次执行后将 `B` 合并进 `A`。
- 首个批次 `A` 为空，不会因为属性塔推进而强制预先重置。
- 可触发但未进入当前候选批次的属性塔节点，不参与 `B` 的计算。
- 不同属性塔在同一累充日内推进不强制重置；同一属性塔在同一累充日内再次推进才强制重置。
- 全属性塔抵达节点本身不推进四个单塔前沿，因此不计入 `B`。若未来要求全属性塔抵达也参与强制累充重置，需要单独定义它的推进集合。

单属性塔每天最多推进 10 层，而当前属性塔相关礼包相邻触发事件至少相隔 25 层。因此同一单塔不可能在同一累充日内连续触发两个相邻事件；上述规则用于保证规划器的抽象累充日模型与游戏机制一致。

当前乐观卡包策略不使用跨 4 点额外推塔能力，因此不需要处理“跨 0 点累充重置”和“跨 4 点属性塔换日同批”的互斥。若未来新增精确跨 4 点属性塔模式，则同一个 2 小时批次不能同时作为跨 0 点累充批次和跨 4 点属性塔批次。

### 7.2 每日累充赠钻

购买规划页按路径内同一累充日累计付费钻计算实际赠钻：

```text
本次赠钻 =
TotalBonusFreeDiamonds(购买后累计付费钻)
- TotalBonusFreeDiamonds(购买前累计付费钻)
```

不同批次共用同一累充日时，累计付费钻持续累加；跨日重置后，新的累充日从 0 开始计算。

为控制计算规模，规划器可以对时间边界、换日位置和拆批组合进行剪枝；剪枝不能把模型退化为固定同日连续购买或固定同批合并。

## 8. 自动补累充

现实购买中，玩家通常会用同批限时包自然凑累充，只在接近下一档每日累充阈值时使用常驻钻石组合包补足缺口。阈值 `x%` 可由用户调节，设为 `0` 即关闭补包。

### 8.1 触发时机

自动补累充只在某个累充日准备跨 0 点重置前结算；如果下一批继续沿用同一个累充日，则不触发补包判断。这样可以把同日后续批次的限时包购买先自然计入累充进度，避免过早用常驻包补一个后续本来会自然达到的阈值。

### 8.2 阈值判定

跨日前执行一次确定性判断，不产生路径分叉：

```text
gap = 下一档累充付费钻 - 当前累计付费钻
if gap <= currentPaid × x%:
    购买常驻钻石组合包，使累计付费钻 >= 下一档
else:
    不补，直接跨日
```

其中 `x%` 为用户可调的补累充阈值（默认 `10%`）。当前累计付费钻越高、缺口占累计的比例越小，越值得补。例如：

- 累计 5000 付费钻，下一档 6000，缺口 1000。1000 <= 5000 × 10% = 500？**否 → 不补**。
- 累计 5800 付费钻，下一档 6000，缺口 200。200 <= 5800 × 10% = 580？**是 → 补**。

阈值越大，补包越激进（更大缺口也补）；阈值越小，补包越保守（只补极小缺口）。玩家可根据自己的常驻包库存和风险偏好调节。

### 8.3 补包规则

- 只使用 `permanentPacks.json` 中非首次的”钻石组合包”。
- 补包附着在跨日前最后一个已购买限时包的批次上，不单独生成展示行。
- 补包不触发限时包，不影响限时包升降档。
- 补包只瞄准下一档每日累充；达到该档后立即停止。
- 补包采用能达到下一累充档的最低花费组合；若花费相同，优先内容价值更高、包数更少的组合。
- 补包花费独立于主预算，不计入约束，仅在结果中展示补累充总开销。阈值 `x%` 天然将单次补包控制在累计付费钻的 10% 以内（通常 ¥200–2000），不会出现失控大额消费。
- 若缺口不满足阈值条件，则不补。

补包组合允许剪枝，不要求穷举所有常驻钻石组合包组合。剪枝应优先保留接近下一累充阈值、花费较低、符合小额补缺口定位的组合。

## 9. 计算触发

规划器不在输入变化时自动重算，需要用户点击“计算购买方案”。这样可以避免多触发源全部启用时，输入框每次变化都触发高成本计算。

## 10. 路径方案类型

页面同时生成多种方案，供玩家按偏好切换：

- 价值最优：在主预算内寻找总评分价值最高的路径。
- 只买小包：只在最低档购买，升档后等待超时掉回最低档。
- 冲最大包：以节省限时包触发机会为目标，优先把可购买次数用于更高档礼包；通常先买入用于升档，到高档后尽量购买高档礼包。必要时可等待超时降档，以保留预算和触发机会给后续更高档购买。

这些方案共用同一套触发机会、批次约束和升降档规则，只是购买动作选择策略不同。

## 11. 展示

购买批次需要保留可展开明细，方便查看每个礼包的触发点、价格、CE、价值和内容物。购买批次不合并；连续不购买的批次可以压缩展示，例如：

```text
第 2-4 批，连续不买 ×3
```

若某批次采用补累充，应在同一个展开区域列出实际使用的非首次常驻钻石组合包。

规划结果汇总应显示：

- 限时包花费（受主预算约束）。
- 补累充总开销（由阈值判定，不计入预算约束）。
- 总花费 = 限时包花费 + 补累充总开销（仅供参考）。
- 总价值。
- 补累充批次数。
- 补累充阈值（当前设定值）。

汇总不显示补包总购买个数；补包总购买个数只适合在补累充批次详情里展示。

列表层优先展示：

- 推进范围。
- 触发节点数 / 购买限时包数。
- 是否跨日重置累充。
- 是否采用补累充。
- 本批花费、价值和下一批档位。

礼包明细、触发节点明细、触发未买原因、补包组合和累充档位解锁信息放在点击展开详情中展示。
