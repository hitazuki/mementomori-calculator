# 超值限时组合包购买规划需求

本文档记录购买规划器需要满足的产品行为和用户可见规则。需求文档在需求核对前不主动改变；算法、性能策略和实现取舍改动写入 `implementation-design.md`。

游戏内稳定规则见 `game-rules.md`。

## 目标

购买规划器在用户给定的预算、当前礼包档位、各触发源当前进度与计划推进范围内，生成一条或多条购买路径。

核心目标：

```text
在限时包花费 <= 主预算、补累充花费 <= 补累充浮动预算的约束下，最大化购买路径的总评分价值
```

礼包内容价值继续使用 `../items-calc-formula.md` 中定义的 CE 评分逻辑。购买规划页中的每日累充赠钻按路径内实际累计付费钻计算，不使用单包 `1.2` 倍估算。

由于同一批内触发礼包越多，玩家执行压力越大，规划器在不降低总价值、预算可行性和升降档可达性的前提下，应优先选择同批触发数更少、批次压力更低的方案。

规划器应允许将可同批触发的机会拆分到不同批次，并将跨累充日重置后获取更多每日累充奖励的方案纳入候选。

若主预算用不完，是因为继续购买限时包会超出主预算，或后续在当前档位没有可购买礼包。

## 输入

页面字段含义：

- 预算（日元）：本次可用于购买限时组合包的主预算。
- 补累充浮动预算比例：独立于主预算的补累充预算，按主预算百分比计算，默认 `5%`。
- 当前礼包档位：现在已经处于的限时组合包档位。玩家当前已经处在更高档时，可以改成对应价格。
- 当前层/等级：该触发源目前已经达到的位置；规划器只考虑大于该值的触发点。
- 计划推到：本次规划中该触发源最多推进到的位置。
- 2小时内最多触发：该触发源在同一个卡包批次内最多能连续触发多少个限时包；这是同批上限，不要求必须合并到上限。

主线的当前进度和计划推到支持 `章节-关卡` 格式，例如 `13-28`；也支持直接填写游戏内连续关卡进度，例如 `336`。其他来源使用数字层数或等级。

页面默认模板使用中等触发区间，减少首次试算压力；这只是输入预设，玩家仍可手动扩大“计划推到”范围。

## 触发批次

主线、玩家等级、无穷塔、各属性塔在规划器中视作独立触发源。

规划器需要支持玩家主动控制触发时机的“乐观卡包”假设：玩家可以控制主线、无穷塔、属性塔等来源的推进节奏，在限定范围内把触发机会安排到一个或多个 2 小时卡包批次中。

在该假设下：

- 同一触发源的触发顺序必须保持，不允许把后面的触发点排到前面的触发点之前。
- “2小时内最多触发”表示同一触发源在同一批内的触发上限；主线和无穷塔通常可大于 1，玩家等级和单个属性塔通常为 1。
- 不同触发源若能通过控进度安排到同一 2 小时窗口，可以进入同一个全局结算批次。
- 可同批不代表必须同批；同一来源未达到上限时也可以主动拆批，不同来源可合批也可以拆到不同批。
- 分批不是单纯的展示拆分。某个触发机会被安排到独立批次后，会改变后续批次开始时各触发源的已推进位置，并影响后续可生成、可合批的触发节点。
- 未进入当前批次的触发节点不视为已触发，可以保留到后续批次；因此规划器可以先通过其他触发节点升档，再在更高档位触发并购买该节点对应礼包。
- 合批不按固定来源批次序号决定。只要触发节点满足各自来源的先后顺序、同批触发上限，以及属性塔/全属性塔等拓扑层级约束，就可以被安排进同一个全局批次。
- 每个全局结算批次统一应用一次升降档规则：批内至少购买 1 个限时包则下一批升档，批内全不买则下一批降档。
- 拆批或合批会改变升降档次数、后续可购买档位、同一累充日内的累计付费钻，以及是否有机会跨 0 点重置每日累充进度。
- 因此批次编排本身属于购买路径选择的一部分，而不是单纯的展示分组。

## 全属性塔抵达

“全属性塔抵达”不是独立填写的触发源，而是由蓝塔、红塔、翠塔、黄塔自动派生。

只有四个属性塔都在规划器中启用，并且四塔共同跨过某个“全属性塔抵达”礼包层级时，该礼包才会进入对应批次。

全属性塔抵达机会不能早于四塔都达到该层。其最早可触发位置由四塔中最慢达到该层的塔决定：

```text
最早可触发位置 = max(蓝塔达到该层的位置, 红塔达到该层的位置, 翠塔达到该层的位置, 黄塔达到该层的位置)
```

满足该约束后，全属性塔抵达机会可以与其他满足同批约束的触发机会合批，也可以拆到独立批次。

因此页面不为“全属性塔抵达”单独提供起始层、目标层或每批触发数。

## 属性塔拓扑

规划器采用乐观卡包假设，但不允许任意打乱属性塔事件顺序。属性塔应按“每个单塔独立前沿 + 全属性塔派生门槛”建立拓扑：

```text
全属性塔抵达 225
单属性塔 250（蓝/红/翠/黄可同批）
全属性塔抵达 275
单属性塔 300（蓝/红/翠/黄可同批）
...
```

规划器不要求用户填写真实日期、开始时间或规划天数。真实星期只影响玩家准备卡点的过程，不改变礼包事件的拓扑顺序。

规划器可以使用抽象时间模型处理批次顺序、2 小时超时、0 点累充重置等时间影响，不要求模拟真实日历。

单属性塔每天最多推进 10 层，而当前属性塔相关触发事件相邻至少相隔 25 层。因此同一个属性塔不可能在同一个累充日内连续触发两个相邻属性塔事件；属性塔拓扑中的相邻事件之间必须视为已经跨过至少一次 0 点累充重置。

同一拓扑层级内的不同属性塔可以分批触发，例如蓝塔 250 与红塔 250 拆成两个批次；这种拆批不要求累充重置。某个单塔触发后，只推进该塔自己的前沿，不会把同层其他塔视为已触发。

例如蓝塔 250 已触发，而红塔/翠塔/黄塔 250 尚未触发时，下一批可触发前沿应同时包含红塔 250、翠塔 250、黄塔 250 和蓝塔 300。只有下一批实际包含蓝塔 300 这类进入后续拓扑层级的节点时，才强制视为跨过累充日；若下一批只触发红塔/翠塔/黄塔 250，则继续共用当前累充日进度。

全属性塔抵达节点由四塔进度共同派生。它不能早于四塔都达到该层；例如全属性塔抵达 275 需要四个单塔前沿都已经越过 250 层，或该层处于四塔当前前沿到下一单塔触发层之间。

只有从一个属性塔拓扑层级进入下一个拓扑层级时，才强制视为跨过累充日。同层拆批、同层合批都不强制重置累充。

当前乐观卡包策略不使用跨 4 点额外推塔能力，因此不需要处理“跨 0 点累充重置”和“跨 4 点属性塔换日同批”的互斥。

若未来新增精确跨 4 点属性塔模式，则同一个 2 小时批次不能同时作为跨 0 点累充批次和跨 4 点属性塔批次。

## 计算触发

规划器不会在输入变化时自动重算，需要用户点击“计算购买方案”。这样可以避免多触发源全部启用时，输入框每次变化都触发高成本计算。

## 每日累充与常驻补包

### 累充日进度

购买规划页按路径内同一天累计付费钻计算实际赠钻：

```text
本次累充赠钻 = TotalBonusFreeDiamonds(购买后累计付费钻) - TotalBonusFreeDiamonds(购买前累计付费钻)
```

不同批次可以共用同一个累充日进度，也可以在批次之间跨过 0 点并重置每日累充进度。是否跨日会影响每日累充奖励，因此应作为购买路径候选的一部分。

每日累充重置只在两个全局批次之间发生；同一个全局批次内的所有限时包购买和补累充购买都视为发生在同一个累充日内，不模拟批内跨 0 点购买顺序。

为控制计算规模，规划器可以对时间边界、换日位置和拆批组合进行剪枝；剪枝不应把模型退化为固定同日连续购买或固定同批合并。

### 补累充预算

补累充使用独立于主预算的小额浮动预算，不应因为限时包主预算已用满而直接放弃高价值补累充。

浮动预算按主预算百分比计算，默认 `5%`。主预算越高，可用于补累充的浮动预算也相应提高。

补累充浮动预算只用于非首次常驻钻石组合包，不会增加可用于购买限时组合包的主预算。

### 自动补包规则

常驻钻石组合包补累充是固定机制，不作为用户可关闭选项。现实购买中，玩家通常会用同批限时包自然凑累充，只在接近下一档每日累充阈值时使用常驻钻石组合包补足缺口；因此规划器必须把可行补累充纳入路径候选。

补累充浮动预算是整条路径共享资源。自动补累充只在某个累充日准备跨 0 点重置前结算；如果下一批继续沿用同一个累充日，则不生成补包候选。这样可以把同日后续批次的限时包购买先自然计入累充进度，避免过早用常驻包补一个后续本来会自然达到的阈值。

若准备跨日前可以补到下一档，但保留浮动预算给后续累充日可能获得更高总价值，规划器可以同时保留“跨日前补累充”和“不补直接跨日”两个候选，由全局路径价值决定最终选择。

当某个累充日即将跨日重置，且该累充日最后一个已购买限时组合包的批次结算后接近下一档每日累充阈值，补累充浮动预算内又可通过非首次常驻钻石组合包补足时，规划器应生成补累充候选：

- 只使用 `permanentPacks.json` 中非首次的“钻石组合包”。
- 补包附着在跨日前最后一个买入批次上，不会单独生成独立展示行。
- 补包不会触发限时包，不会影响限时包升降档，但会消耗补累充浮动预算并改变后续路径选择。
- 补包只瞄准本批限时包购买后的下一档每日累充；达到该档后立即停止，即使补累充浮动预算还有剩余也不会多买常驻包。
- 补包只处理补累充浮动预算内的小额缺口；超过该范围视为额外购买大额常驻包，不作为自动补累充。
- 补包采用能达到下一累充档的最低花费组合；若花费相同，优先内容价值更高、包数更少的组合。

### 剪枝与例外

- 补包组合允许剪枝，不要求穷举所有常驻钻石组合包组合。剪枝应优先保留接近下一累充阈值、花费较低、符合小额补缺口定位的组合。
- 当补累充浮动预算不足、缺口超过补包剪枝范围，或没有可用的非首次常驻钻石组合包组合时，只保留不补累充候选。
- 当最终方案跳过了某个可补累充批次，应是因为补累充浮动预算被保留给后续更高价值路径，或该补累充候选在全局搜索中被剪枝淘汰。

### 展示

- 页面应允许查看和调整补累充浮动预算比例，并根据主预算和浮动预算比例展示补累充预算与总预算。
- 页面按批次展示最终方案实际采用的补包，例如 `第 1 批：钻石组合包 80 ×2`，不展示未被采用的补包候选。
- 页面应分别展示限时包花费、补累充花费、补累充预算剩余和总花费。

例如限时包累计到 5,900 付费钻时，可以用两个 80 付费钻常驻包补到 6,060 付费钻，从而解锁 6,000 档每日累充奖励。

## 多方案展示

页面会同时生成多种方案，供玩家按偏好切换：

- 价值最优：在主预算和补累充浮动预算内寻找总评分价值最高的路径。
- 只买小包：只在最低档购买，升档后等待超时掉回最低档。
- 冲最大包：以节省限时包触发机会为目标，优先把可购买次数用于更高档礼包；通常先买入用于升档，到高档后尽量购买高档礼包。
- 冲最大包方案可以酌情支持等待超时降档：当继续购买当前批次会消耗有限触发机会、降低后续购买更高档礼包的能力，或预算更适合后续高档购买时，允许选择不买当前批次并降档。

这些方案共用同一套触发机会、批次约束和升降档规则，只是购买动作选择策略不同。

## 展示

购买批次需要保留可展开明细，方便查看每个礼包的触发点和内容物。

连续不购买的批次可以压缩展示，例如：

```text
第 2-4 批，连续不买 ×3
```

购买批次不合并，因为用户需要展开查看每个礼包的触发点、价格、CE、价值和内容物。若该批次采用了补累充，应在同一个展开区域列出实际使用的非首次常驻钻石组合包。

规划结果汇总应显示补累充批次数，而不是补包总购买个数。补包总购买个数只适合在补累充批次详情里展示。

路径表格应避免“触发”和“操作”两列重复表达。列表层优先展示简单可扫描字段：

- 推进范围。
- 触发节点数 / 购买限时包数。
- 是否跨日重置累充。
- 是否采用补累充。
- 本批花费、价值和下批档位。

礼包明细、触发节点明细、触发未买原因、补包组合和累充档位解锁信息放在点击展开详情中展示。
